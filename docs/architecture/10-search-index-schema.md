# Search Index Schema

## Overview

The Azure AI Search index schema is defined programmatically using the `Azure.Search.Documents` SDK's `FieldBuilder` and attribute-based mapping. The index is created or updated at the start of each reindex operation.

## Index Configuration

| Setting | Value |
|---------|-------|
| Index name | `rag-navigator-index` (configurable) |
| Vector algorithm | HNSW (Hierarchical Navigable Small World) |
| Vector metric | Cosine similarity |
| Vector dimensions | 1536 (matches text-embedding-ada-002 / text-embedding-3-small) |
| Keyword analyzer | `en.lucene` (English-optimized) |

## Field Definitions

| Field | Type | Key | Searchable | Filterable | Sortable | Facetable | Vector |
|-------|------|-----|------------|------------|----------|-----------|--------|
| `ChunkId` | `Edm.String` | Yes | No | Yes | No | No | No |
| `DocumentId` | `Edm.String` | No | No | Yes | No | No | No |
| `DocumentTitle` | `Edm.String` | No | Yes | No | No | No | No |
| `FileName` | `Edm.String` | No | No | Yes | No | Yes | No |
| `Section` | `Edm.String` | No | Yes | No | No | No | No |
| `ChunkIndex` | `Edm.Int32` | No | No | Yes | Yes | No | No |
| `Content` | `Edm.String` | No | Yes (en.lucene) | No | No | No | No |
| `ContentVector` | `Collection(Edm.Single)` | No | No | No | No | No | Yes |

## Field Design Rationale

### ChunkId (Key)
Deterministic composite key: `{documentId}_chunk_{index}`. This makes reindexing idempotent — the same document produces the same chunk IDs.

### DocumentId (Filterable)
A SHA-256 hash of the file name, truncated to 12 hex characters. Used to group chunks belonging to the same source document. Filterable to support per-document queries (e.g., "show all chunks from this document").

### DocumentTitle (Searchable)
The title extracted from the first `#` heading in the document. Searchable so that queries like "onboarding guide" match the title even if the content chunk doesn't contain those exact words.

### FileName (Filterable, Facetable)
The original file name. Filterable for scoped queries. Facetable to support a document-count sidebar (not currently used but ready for future features).

### Section (Searchable)
The nearest `##` or `###` heading above the chunk. Searchable so that section names contribute to keyword relevance (e.g., searching for "failover" matches both content and section headings).

### ChunkIndex (Filterable, Sortable)
The sequential position of the chunk within its document. Sortable to enable ordered reconstruction of a document's content if needed. Filterable for range queries.

### Content (Searchable, en.lucene)
The main text of the chunk. Uses the `en.lucene` analyzer for English-optimized tokenization, stemming, and stop-word handling. This is the primary field for BM25 keyword search.

### ContentVector (Vector, 1536 dimensions)
The embedding vector generated by Azure OpenAI. Configured with:
- **HNSW algorithm:** Efficient approximate nearest-neighbor search, balancing speed and recall.
- **Cosine metric:** Standard for text embeddings; measures directional similarity regardless of vector magnitude.
- **HNSW parameters:** `M=4`, `efConstruction=400`, `efSearch=500`. These provide good recall for a small-to-medium index. For larger indexes, increasing `M` improves recall at the cost of memory and indexing time.

## Vector Search Configuration

```json
{
  "vectorSearch": {
    "algorithms": [{
      "name": "hnsw-config",
      "kind": "hnsw",
      "hnswParameters": {
        "metric": "cosine",
        "m": 4,
        "efConstruction": 400,
        "efSearch": 500
      }
    }],
    "profiles": [{
      "name": "vector-profile",
      "algorithm": "hnsw-config"
    }]
  }
}
```

## Hybrid Search Behavior

When a query is executed, Azure AI Search runs both retrieval methods in parallel:

1. **BM25 keyword search** on `Content`, `DocumentTitle`, and `Section` fields.
2. **HNSW vector search** on `ContentVector` using the query embedding.

Results are merged using **Reciprocal Rank Fusion (RRF)**:
- Each method produces a ranked list.
- RRF score = Σ(1 / (k + rank_i)) across all methods where the document appears.
- This produces a single merged ranking without requiring score normalization.

## Schema Evolution

The current schema is designed for the demo's scope. Production enhancements:

| Enhancement | Change | Purpose |
|-------------|--------|---------|
| Semantic ranker | Add `SemanticConfiguration` | L2 re-ranking for better relevance |
| Access control | Add `AccessGroup` filterable field | Per-document RBAC filtering |
| Timestamps | Add `IndexedAt` field | Track freshness, enable incremental updates |
| Categories | Add `Category` facetable field | Filter by document type (ADR, runbook, etc.) |
| Scoring profile | Custom `ScoringProfile` | Boost recent documents or specific categories |
